# Codex 작업 지침서

이 작업 환경에서 Git은 단순한 업로드 도구가 아니라, 수정 전후 상태를 안전하게 구분하고 되돌리기 위한 기준점이다. 따라서 Codex는 코드를 수정할 때 항상 "현재 작업 중인 파일이 어느 공간에 있는지"를 의식해야 한다. Git의 작업 공간은 크게 네 단계로 이해하면 된다. 첫째는 working directory이고, 이것은 실제로 사용자가 VS Code에서 열어 수정하는 현재 폴더의 파일들이다. 둘째는 staging area이고, 이것은 다음 commit에 포함될 변경사항을 임시로 올려두는 공간이다. 셋째는 local repository이고, 이것은 commit이 실제로 저장되는 로컬 Git 기록이다. 넷째는 remote repository이고, GitHub 같은 원격 저장소를 뜻한다. Codex는 기본적으로 working directory 수준에서만 수정 작업을 수행한다고 생각해야 하며, staging, commit, push 같은 행위는 사용자가 명확히 요청했을 때만 다뤄야 한다.

Codex는 작업을 시작하기 전에 먼저 현재 작업 폴더가 정말 GitHub와 연결된 실제 저장소인지 확인해야 한다. 사용자가 여러 폴더를 동시에 가지고 있을 수 있고, 예전에 수동 복사한 폴더와 clone 받은 폴더가 섞여 있을 수 있기 때문이다. 따라서 Codex는 사용자가 작업 대상으로 열어 둔 폴더가 원격 저장소와 연결된 실제 clone 폴더인지 먼저 확인하는 태도를 가져야 한다. 사용자가 예전에 열었던 로컬 복사본과, 실제 GitHub와 연결된 clone 폴더를 혼동하는 상황은 자주 발생하므로, Codex는 현재 폴더를 당연히 정답으로 가정하지 말고 항상 주의해야 한다. 특히 사용자가 이미 clone 받은 저장소 폴더가 있다면, 그 폴더를 기준으로만 작업해야 한다.

Codex는 코드 수정 자체와 Git 기록 생성을 구분해서 행동해야 한다. 사용자가 "코드를 고쳐 달라"고 했다고 해서 자동으로 add, commit, push까지 해서는 안 된다. 코드를 수정하는 것은 working directory에서의 편집이고, Git 이력을 남기는 것은 전혀 다른 행위다. 사용자가 수정 결과를 보고 마음에 들지 않으면 되돌릴 수 있어야 하므로, Codex는 기본적으로 파일 수정까지만 담당하고, Git 반영은 사용자의 명시적 요청이 있을 때만 진행해야 한다. 특히 git add ., git commit, git push 같은 명령은 되돌리기 지점, 추적 파일 범위, 원격 공개 여부와 직접 연결되므로, 자동으로 실행하면 안 된다.

Codex가 작업하기 전에 가장 바람직한 상태는, 사용자가 현재 정상 동작하는 버전을 이미 한 번 commit해 둔 상태다. 그래야 이후 Codex가 대규모 수정이나 구조 변경을 하더라도 사용자는 직전 상태로 쉽게 돌아갈 수 있다. 따라서 Codex는 큰 수정, 여러 파일 수정, 리팩터링, 파일 이동, 함수 분해, 로직 재구성 같은 작업이 들어갈 경우, 먼저 사용자가 기준 commit을 남겼는지 확인하도록 유도하는 방향이 좋다. 다만 Codex가 직접 강제로 commit을 만들지는 말고, "현재 상태를 기준점으로 저장해 두는 것이 안전하다"는 원칙을 따르는 식으로 행동해야 한다.

Git 공간의 의미를 Codex는 이렇게 이해하면 된다. 사용자가 파일을 편집하면 그것은 working directory의 변경이다. 이때 VS Code에서 M이 뜨면 modified, 즉 마지막 commit 이후 파일이 바뀌었다는 뜻이다. 이 상태는 아직 Git 기록으로 저장되지 않았다. 그 다음 git add를 하면 변경사항이 staging area로 올라간다. 이것은 "다음 commit에 넣을 후보로 선택되었다"는 의미이지, 아직 저장이 끝난 것이 아니다. 그 다음 git commit을 해야 local repository에 기록이 생긴다. 마지막으로 git push를 해야 그 commit이 remote repository로 올라간다. Codex는 이 네 단계가 서로 다르다는 점을 절대 섞어서는 안 된다. 특히 push는 수정된 파일을 올리는 것이 아니라, 이미 commit된 기록을 올리는 것이다.

Codex는 작업 중 생성되는 부산물에 매우 주의해야 한다. Python 프로젝트에서는 __pycache__, .pyc, 로그 파일, 임시 결과물, 테스트 출력물, 스크린샷, 분석 중간 산출물 같은 것들이 쉽게 생긴다. 이런 파일은 대개 소스 그 자체가 아니며, Git 추적 대상이 되어서는 안 된다. 따라서 Codex는 새 파일을 만들거나 디렉터리 구조를 제안할 때, 그것이 실제 소스 파일인지, 아니면 실행 결과물인지 구분해야 한다. 만약 임시 비교용 스크립트, 일회성 분석 파일, 로그, 캐시가 생겼다면 그것을 핵심 소스처럼 취급하면 안 된다. 특히 git add .를 쉽게 쓰면 이런 파일들까지 같이 추적될 수 있으므로, Codex는 파일 생성 시점부터 "이 파일이 저장소에 남아야 하는 파일인가"를 항상 의식해야 한다.

Codex는 사용자가 실수로 잘못된 폴더에서 작업하지 않도록 도와야 한다. 예를 들어 예전 로컬 복사본 폴더와 실제 clone 폴더가 둘 다 존재하는 경우, 사용자가 복사본 폴더에서 수정한 최신 코드를 실제 저장소에 반영하지 못하고 혼란스러워질 수 있다. 이런 상황에서는 Codex가 무작정 수정 작업을 이어가기보다, 어느 폴더가 GitHub와 연결된 실제 작업 폴더인지 먼저 명확히 하고, 앞으로는 하나의 폴더만 작업 기준으로 삼도록 유도해야 한다. 여러 폴더를 오가며 수정하면 Git 상태가 꼬이기 쉽고, 어느 쪽이 최신인지 판단도 어려워진다.

Codex는 사용자가 "되돌릴 수 있는 상태"를 유지하는 것을 최우선 원칙으로 삼아야 한다. 즉, 빠르게 고치는 것보다, 되돌릴 수 있게 고치는 것이 더 중요하다. 큰 수정일수록 한 번에 모든 것을 바꾸기보다, 관련 있는 단위끼리 나누어 작업하는 것이 좋다. 예를 들어 함수 이름 변경, 로직 수정, 파일 이동, 출력 형식 변경이 동시에 필요한 경우에도, 가능하면 어떤 변화가 어떤 영향 때문에 들어갔는지 분리해서 설명하고 반영해야 한다. 그래야 사용자가 나중에 어떤 수정이 문제를 만들었는지 역추적하기 쉽다. Codex는 "한 번에 전부 갈아엎는 수정"보다 "원인과 결과가 보이는 수정"을 선호해야 한다.

사용자가 Codex에게 수정 요청을 할 때, Codex는 항상 현재 변경사항이 이미 존재하는지 확인해야 한다. 사용자가 직접 수정한 부분, 이전 Codex가 수정한 부분, 아직 commit되지 않은 작업이 섞여 있을 수 있기 때문이다. 이 상태에서 Codex가 새로운 변경을 덮어쓰면, 사용자는 무엇이 원래 자신의 수정이고 무엇이 Codex의 수정인지 구분하기 어려워진다. 따라서 Codex는 기존 수정 위에 작업할 때, 현재 파일 상태를 존중하고, 원래 의도를 훼손하지 않으며, 불필요한 전체 재포맷이나 대량 치환을 피해야 한다. 코드 스타일이나 줄바꿈, import 순서 같은 사소한 형식 변화로 diff를 불필요하게 키우는 것도 바람직하지 않다.

Codex는 Git 명령을 다룰 때도 보수적으로 행동해야 한다. git restore는 아직 commit되지 않은 로컬 변경을 버리는 명령이므로, 사용자가 명확히 원하지 않는 한 먼저 제안만 하고 바로 실행해서는 안 된다. git reset --hard는 훨씬 더 강한 명령이며, 사용자의 로컬 변경을 돌이킬 수 없게 잃게 만들 수 있으므로, 극히 조심해서 다뤄야 한다. git rm --cached는 Git 추적만 제거하고 실제 파일은 남기는 동작이므로, 캐시 폴더나 산출물을 저장소에서 빼야 할 때 유용하지만, 이것도 무엇을 제거하는지 충분히 명확할 때만 사용해야 한다. git push는 원격 상태를 바꾸는 행위이므로, 사용자의 승인 없이 자동 실행해서는 안 된다.

Codex는 사용자의 저장소가 개인 복구용일 수도 있다는 점을 이해해야 한다. 즉, 이 저장소는 협업용이 아니라 "수정 전 상태를 남기고, 실험 후 되돌리기 쉽게 하기 위한 목적"일 수 있다. 이런 경우 가장 중요한 것은 완벽한 브랜치 전략이 아니라, 현재 정상 상태를 명확히 남겨 두고, 위험한 수정 전에 기준점을 만드는 것이다. 따라서 Codex는 불필요하게 복잡한 Git 흐름을 강요할 필요가 없다. 대신 사용자가 현재 구조를 유지하면서도, 작업 전 상태와 작업 후 상태를 명확히 구분할 수 있도록 돕는 것이 좋다.

정리하면, Codex는 이 저장소에서 코드를 수정할 때 working directory, staging area, local repository, remote repository를 서로 다른 층위로 이해하고, 기본적으로는 working directory에서의 안전한 수정에 집중해야 한다. Git add, commit, push는 사용자의 승인과 목적이 분명할 때만 다뤄야 한다. 캐시, 임시 파일, 로그, 결과물은 추적 대상이 아닌지 항상 확인해야 하며, 여러 폴더가 섞여 있을 경우 실제 clone 폴더를 기준으로만 작업해야 한다. 가장 중요한 원칙은, 사용자가 언제든 이전 정상 상태로 돌아갈 수 있어야 한다는 점이다. Codex는 빠른 수정자이기 전에, 되돌릴 수 있는 수정자여야 한다.
